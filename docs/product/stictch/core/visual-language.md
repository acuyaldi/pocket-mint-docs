# Pocket Mint Visual Language

Version: 1.0

Status: Foundation

Owner: Pocket Mint

Category: Visual Language

---

# Purpose

This document defines the visual language used across Pocket Mint.

It establishes a consistent appearance for every interface regardless of screen or platform.

Visual language should communicate professionalism before personality.

Every visual decision should reinforce Pocket Mint as a Private Financial Workspace.

---

# Visual Philosophy

Pocket Mint is intentionally quiet.

The interface should support financial understanding.

Never compete with it.

Visual language should reduce cognitive effort.

Users should notice information before interface decoration.

---

# Design Personality

Pocket Mint should feel:

Professional

Calm

Minimal

Organized

Trustworthy

Timeless

Confident

Pocket Mint should never feel:

Playful

Flashy

Aggressive

Overdesigned

Experimental

Gamified

Marketing-driven

---

# Surface Philosophy

Use surfaces to organize information.

Not decorate the interface.

Every surface should communicate purpose.

Avoid unnecessary layers.

Avoid floating elements without meaning.

---

# Card Philosophy

Cards are the primary information container.

Cards should:

Group related information.

Separate logical sections.

Support readability.

Cards should never become decorative objects.

---

# Card Appearance

Cards should appear:

Clean

Structured

Subtle

Lightweight

Use soft corners.

Use restrained elevation.

Prefer borders over heavy shadows.

Cards should feel stable.

Not floating.

---

# Corner Radius

Use consistent corner radius throughout the application.

Avoid mixing multiple radius styles.

Rounded corners should communicate softness.

Not playfulness.

---

# Borders

Borders define separation.

Not decoration.

Prefer subtle borders over heavy shadows.

Every border should improve readability.

---

# Shadows

Shadows communicate elevation.

Not decoration.

Use the minimum shadow necessary.

Avoid dramatic floating effects.

Avoid layered shadow stacks.

---

# Elevation

Pocket Mint uses limited elevation.

Hierarchy should come from composition.

Not shadows.

Elevated surfaces should communicate interaction.

Static content should remain grounded.

---

# Color Usage

Colors communicate meaning.

Never decoration.

Primary color

Brand identity.

Success

Positive financial state.

Warning

Requires attention.

Danger

Negative financial state.

Neutral

Supporting information.

Every color must have semantic meaning.

---

# Financial Color Semantics

Positive

Assets

Income

Completed

Negative

Debt

Expense

Overdue

Neutral

Labels

Descriptions

Metadata

Information

Warning

Upcoming obligations

Attention

Review required

Never invert financial semantics.

---

# Iconography

Icons support recognition.

Icons never replace labels.

Use one icon style consistently.

Avoid mixing:

Outline

Filled

Rounded

Sharp

Icons should remain visually secondary.

---

# Illustration Policy

Illustrations are optional.

Use illustrations only for:

Empty States

Onboarding

Landing Page

Never use illustrations to decorate financial information.

Dashboard should remain information-first.

---

# Imagery

Pocket Mint should not rely on photography.

Illustrations should be simple.

Abstract.

Supportive.

Never emotional.

---

# Typography Relationship

Typography communicates hierarchy.

Visual styling should never overpower typography.

Numbers should remain the strongest visual element.

---

# Financial Numbers

Financial values are the primary visual language.

Large numbers communicate importance.

Supporting labels remain visually quieter.

Never decorate financial values.

---

# Empty States

Empty states should reassure users.

Not entertain them.

Examples:

No Wallets

No Transactions

No Installments

Illustrations should remain subtle.

Copy should encourage action.

---

# Loading States

Loading states should preserve layout.

Prefer skeleton loading.

Avoid centered spinners for entire pages.

Loading should feel progressive.

---

# Motion

Motion communicates change.

Not delight.

Animations should be:

Short

Subtle

Purposeful

Avoid decorative animations.

Avoid bouncing effects.

Avoid exaggerated transitions.

---

# Visual Consistency

Every screen should appear as though designed at the same time.

Avoid introducing:

Unique card styles

Unique shadows

Unique icon systems

Unique radius

Unique button styles

Consistency builds trust.

---

# Density

Pocket Mint follows a balanced visual density.

Avoid:

Visual emptiness.

Visual overload.

Interfaces should feel breathable.

Never sparse.

Never crowded.

---

# Branding

Brand identity should remain subtle.

Brand color supports navigation.

Not attention.

Financial information always has higher priority than branding.

---

# Decorative Elements

Decorative elements should be minimal.

Avoid:

Gradients without purpose

Large abstract shapes

Decorative backgrounds

Glowing effects

Glassmorphism

Neumorphism

Excessive blur

Visual language should remain timeless.

---

# Responsive Consistency

Desktop

Tablet

Mobile

should all feel like the same product.

Only layout changes.

Visual language never changes.

---

# Anti Patterns

Do not:

Overuse shadows

Overuse gradients

Overuse borders

Mix icon styles

Mix button styles

Create decorative cards

Use colorful backgrounds for information

Use oversized illustrations

Compete with financial information

Visual language should always support clarity.

---

# Acceptance Criteria

Visual language is successful when:

✓ Every screen feels like the same product.

✓ Financial information dominates visual attention.

✓ Cards feel consistent.

✓ Elevation is subtle.

✓ Icons remain supportive.

✓ Colors communicate meaning.

✓ Decorative elements remain restrained.

✓ Branding never competes with content.

---

# Definition of Done

The visual language is complete when every Pocket Mint interface shares the same visual identity regardless of screen or platform.

This document serves as the authoritative visual reference for all future UI generation and frontend implementation.
